# plan-v2 → v3 변경 검증 결과 (SEO 콘텐츠 HTML → JSON 전환)

> 검증일: 2026-03-31
> 검증 대상: plan-v2.md 기반 HTML→JSON 전환 설계
> 변경 범위: 8개 태스크 수정 (T04, T05, T12, T13-a, T13-c, T13-d, T14, T16, T25), 보안/리스크/에러복구 섹션 업데이트

## 1. 전제 검증

| # | 암묵적 전제 | 근거 | 틀릴 경우 파급도 |
|---|---|---|---|
| P1 | gpt-4o-mini가 지정된 JSON 스키마를 안정적으로 출력한다 | OpenAI `response_format={"type":"json_object"}`를 사용하면 유효 JSON 출력이 보장되지만, **스키마 준수**(필드명, 배열 길이 등)는 보장되지 않음 | **중** — 파싱은 성공하나 스키마 미준수 시 재생성 비용 발생. jsonschema.validate로 잡히므로 데이터 오염은 없으나, 재시도 비율이 5%를 넘으면 비용/시간 증가 |
| P2 | 6개 JSON 필드가 SEO 관점에서 HTML 자유 형식과 동등한 가치를 가진다 | 구글은 최종 렌더링된 HTML을 평가하므로 저장 형식(JSON vs HTML)은 무관. 단, 필드 분리로 인해 문단 간 자연스러운 흐름이 약해질 수 있음 | **하** — 템플릿 설계로 보완 가능 |
| P3 | JSONB 컬럼이 Supabase Free 티어의 500MB 한도 내에서 운영 가능하다 | JSONB 오버헤드 포함 ~36MB. 한도(500MB) 대비 충분한 여유 | **하** |
| P4 | React JSX의 자동 이스케이핑이 XSS를 완전히 방어한다 | React의 `{text}` 렌더링은 HTML 엔티티를 자동 이스케이핑. React 공식 보안 설계 원칙 | **하** |
| P5 | T13-d(파일럿 배포, 3주차)가 T16(프론트엔드 템플릿, 4주차) 없이 JSON을 렌더링할 수 있다 | Python 스크립트로 간단한 HTML 변환 가능. 100건 파일럿이므로 간단한 스크립트로 충분 | **하** |

## 2. 실행 가능성

- **비현실적 일정 구간**: 해당 없음. 순 시간 영향 약 -35분(절감).
- **미확보 역량/자원**: jsonschema(학습 ~15분), OpenAI JSON mode(파라미터 1개 추가). 학습 비용 무시할 수준.
- **병목 의존성**: 해당 없음. 외부 의존성이 줄어드는 방향의 변경.

## 4. 최악의 시나리오 3가지

### 시나리오 1: AI JSON 스키마 준수율 저조
- **발생 확률**: 낮음 | **피해 규모**: 중
- **대응**: 프롬프트에 예시 JSON + `response_format=json_object` + 1회 재시도

### 시나리오 2: JSON 스키마 변경 빈발
- **발생 확률**: 낮음 | **피해 규모**: 중
- **대응**: 보수적 스키마 설계 + JSONB `jsonb_set()` 부분 업데이트 가능

### 시나리오 3: 필드 분리로 SEO 품질 저하
- **발생 확률**: 낮음 | **피해 규모**: 높음
- **대응**: T13-d 파일럿 100건에서 최종 렌더링 품질 검토. 프롬프트 조정.
