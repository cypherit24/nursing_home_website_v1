## 1. 전제 검증 해결방안

### P1: B550928 API Rate Limit 미확인

**문제**: 2만 건 비동기 호출 시 API Rate Limit 정책·일일 호출 제한 수치 미확인

**해결방안**:
1. **사전 확인**: 공공데이터포털 B550928 API 상세페이지에서 일일 트래픽 제한 확인 (일반적으로 일 1,000회 기본, 신청 시 확장 가능)
2. **트래픽 확장 신청**: 공공데이터포털에서 "활용신청" 시 트래픽 상향 요청 (무료, 최대 일 10,000~50,000회 가능)
3. **방어적 호출 설계**:
   - Semaphore(10) → **Semaphore(5)**로 보수적 조정
   - 429/503 응답 시 지수 백오프 (5초→15초→45초→135초)
   - 일일 제한 도달 시 자동 중단 + 다음날 이어서 실행 (IS NULL 조건 활용)
4. **분할 실행**: 2만 건을 4일로 분할 (일 5,000건) — GitHub Actions scheduled 워크플로우에서 offset 파라미터로 제어

**비용**: 없음 (트래픽 확장 신청 무료)

### P2: OpenAI gpt-4o-mini 비용 산출 근거 부재

**문제**: 토큰 단가·평균 출력 길이 기반 산출 근거 없음

**해결방안 — 구체적 비용 산정**:

| 항목 | 수치 |
|---|---|
| gpt-4o-mini 입력 단가 | $0.15 / 1M tokens |
| gpt-4o-mini 출력 단가 | $0.60 / 1M tokens |
| 건당 평균 입력 (프롬프트+시설 데이터) | ~500 tokens |
| 건당 평균 출력 (SEO HTML 글) | ~800 tokens |
| 총 건수 | 20,000건 |

**산출**:
- 총 입력: 20,000 × 500 = 10,000,000 tokens → 10M × $0.15/1M = **$1.50**
- 총 출력: 20,000 × 800 = 16,000,000 tokens → 16M × $0.60/1M = **$9.60**
- **합계: ~$11.10** (plan-v1의 $10~15 추정은 적절)

**추가 안전장치**:
- OpenAI 계정에 **월 $20 하드 리밋** 설정 (Settings > Billing > Usage limits)
- TEST_MODE 3건 실행 후 실제 토큰 사용량 측정 → 전체 비용 재산정 후 실행 승인

**비용**: ~$11.10 (1회성)

### P3: Vercel 무료 티어 2만 페이지 SSG 빌드 제한

**문제**: Vercel Hobby 플랜의 빌드 시간 45분 제한에 2만 페이지 SSG가 걸릴 수 있음

**해결방안 — Vercel Pro 업그레이드 권고**:

| 항목 | Hobby (무료) | Pro ($20/월) |
|---|---|---|
| 빌드 시간 제한 | 45분 | 45분 |
| 동시 빌드 | 1개 | 1개 |
| Serverless Function 실행시간 | 10초 | 60초 |
| ISR | 지원 | 지원 |
| 대역폭 | 100GB/월 | 1TB/월 |
| Edge Middleware | 지원 | 지원 |

**핵심**: Hobby와 Pro 모두 빌드 시간 45분 제한이 동일하므로, **업그레이드만으로는 해결 불가**. 아키텍처 변경이 필요하다.

**구체적 전략 — SSG + ISR 하이브리드**:
1. **빌드 시 정적 생성**: 상위 1,000건만 (인구 밀집 지역 우선)
2. **나머지 19,000건**: `dynamicParams = true` + ISR `revalidate: 2592000` (30일)
   - 첫 방문 시 서버에서 렌더링 → 캐싱 → 이후 정적 서빙
3. `generateStaticParams`에서 Supabase 조회 시 `ORDER BY sido_population DESC LIMIT 1000`

**예상 빌드 시간**: 1,000페이지 × ~1초/페이지 = ~17분 (45분 제한 내 충분)

**비용 판단**:
- Hobby 플랜으로 시작 가능 (ISR 하이브리드 전략 적용 시)
- 트래픽이 월 100GB 초과 시 → **Vercel Pro $20/월** 업그레이드
- 초기에는 $0, 트래픽 성장 후 $20/월

### P5: xlsx 컬럼 구조 변경 가능성

**문제**: 컬럼 매핑 하드코딩, 포털 파일 구조 변경 시 파이프라인 오류

**해결방안**:
1. **헤더 기반 매핑으로 전환**: 인덱스(A, B, G) 대신 컬럼 헤더명으로 매핑
   ```python
   COLUMN_MAP = {
       "장기요양기관기호": "facility_code",
       "기관명": "name",
       "시도": "sido",
       "시군구": "sigungu",
       # ...
   }
   df = df.rename(columns=COLUMN_MAP)
   ```
2. **필수 컬럼 검증 추가**: 파싱 시작 전 필수 컬럼 존재 여부 확인
   ```python
   REQUIRED = ["장기요양기관기호", "기관명", "시도", "시군구"]
   missing = set(REQUIRED) - set(df.columns)
   if missing:
       raise ValueError(f"필수 컬럼 누락: {missing}")
   ```
3. **변경 감지 알림**: step1 실행 시 기존 컬럼 목록과 비교 → 차이 있으면 GitHub Issue 자동 생성

**비용**: 없음

### P6: dangerouslySetInnerHTML XSS 위험

**문제**: AI 생성 HTML을 nh3 정화 후 렌더링하지만, 정화 실패 시 XSS 가능

**해결방안**:
1. **이중 방어**: step3(파이프라인)에서 nh3 정화 + 프론트엔드에서 추가 정화
   - Python 측 (step3): `nh3.clean(html, tags={"h1","h2","h3","p","ul","ol","li","strong","em"})` — 허용 태그 화이트리스트
   - Next.js 측: `sanitize-html` 라이브러리로 렌더링 직전 재정화
2. **저장 시점 검증**: DB에 저장 전 정화된 HTML에 `<script>`, `on*=` 속성, `javascript:` URL이 없는지 정규식 검증
3. **CSP 헤더 설정**: Next.js middleware에서 `Content-Security-Policy: script-src 'self'` 설정 — 인라인 스크립트 실행 원천 차단

**비용**: 없음

### N1: 에러 복구 / 부분 재실행 전략

**해결방안**:
1. **원자적 업데이트**: 각 기관의 데이터를 하나의 트랜잭션으로 처리

### N4: SEO 콘텐츠 품질 관리

**문제**: 2만 건 AI 생성 글의 품질 편차, 구글 Helpful Content Update 위험

**해결방안**:
1. **품질 기준 정의**:
   - 최소 길이: 300자 이상
   - 필수 키워드 포함: 기관명, 지역명, "요양원"

   ```python
   def validate_seo(html: str, facility: dict) -> bool:
       text = strip_tags(html)
       if len(text) < 300: return False
       if facility["name"] not in text: return False
       if facility["sido"] not in text: return False
       return True
   ```
3. **실패 시 재생성**: 검증 실패 건은 프롬프트 변형하여 1회 재생성 시도

5. **시설별 고유 데이터 강조**: 프롬프트에 해당 시설의 정원, 현원, 인력, 비급여 항목 등 구체적 수치를 포함시켜 AI 글이 시설마다 실질적으로 다른 정보를 담도록 설계
